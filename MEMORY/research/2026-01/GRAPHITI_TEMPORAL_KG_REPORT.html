1. Key Concepts

What is a Temporal Knowledge Graph?

A temporal knowledge graph extends regular knowledge graphs by adding time as a first-class dimension, enabling:

  • Historical queries: "What did we know about Project X in Q3?"
  • Change tracking: "How has Sarah's role evolved?"
  • Contradiction detection: Automatically invalidate outdated facts
  • Point-in-time reconstruction: "What was the knowledge state on January 15th?"

Bi-Temporal Data Model

Graphiti tracks four time dimensions for every relationship:

Dimension Field Purpose Example
Valid Time valid_at When fact became true in real world "Mike joined Anthropic on 2025-03-15"
Transaction Time created_at When fact was recorded in system Episode ingested at 2026-01-21T10:30:00Z
Invalidation Time invalid_at When fact ceased to be true "Mike left Anthropic on 2026-01-10"
Expiration Time expired_at When superseded by new info System detected contradicting fact

Why This Matters for PAI

Key Insight: Traditional RAG treats knowledge as static. Graphiti's temporal model enables PAI to understand how knowledge evolves - critical for tracking people, projects, and relationships over time.

PRIMARY TUNING LEVER: Docstrings as Prompts

Critical Insight: Pydantic model docstrings are passed directly to the LLM as extraction guidance. More detailed docstrings = better extraction accuracy.
class Organization(BaseModel): """ A company, team, division, or department mentioned in captures. ═══════════════════════════════════════════════════════════════════════════ MANDATORY EXTRACTION RULES - ALWAYS type as Organization: ═══════════════════════════════════════════════════════════════════════════ 1. ANY phrase ending in "team" → Organization - "DevOps team" → Organization(name="DevOps team") - "infrastructure team" → Organization - "QA team" → Organization 2. ANY phrase ending in "division" → Organization - "Platform division" → Organization(name="Platform division") - "Research division" → Organization 3. Department names → Organization - "Legal", "Engineering", "Sales", "HR", "Finance", "Marketing" - "from Legal" → extract "Legal" as Organization ═══════════════════════════════════════════════════════════════════════════ EXAMPLES TO EXTRACT: ═══════════════════════════════════════════════════════════════════════════ "The DevOps team reports to Platform division" → Organization(name="DevOps team", org_type="team") → Organization(name="Platform division", org_type="division") """ org_type: Optional[str] = Field(description="Type: company, team, division, department")

Pattern That Worked (100% Type Accuracy)

  1. MANDATORY EXTRACTION RULES section with explicit patterns
  2. Numbered rules with specific examples
  3. DISAMBIGUATION section (e.g., Project vs Concept)
  4. EXAMPLES TO EXTRACT with input → output format
  5. Field() descriptions with possible values

5. Known Limitations

LLM Extraction Variance

Same input can produce different extractions on different runs.

Mitigation: Detailed docstrings, post-processing validation, deterministic caching.

Node Attribute Versioning (Issue #1166)

Entity properties are destructively overwritten. Historical attribute values lost.

Mitigation: Store history in edges, not node properties.

Cost Per Ingestion

~10-20 LLM calls per episode (~$0.10-0.50 per 1000 words).

Mitigation: Bulk ingestion, smaller models for classification.

Query Complexity

Temporal queries require understanding bi-temporal predicates.

Mitigation: Build query helper functions.
